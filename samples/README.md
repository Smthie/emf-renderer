# Samples

EMF files are grouped by directory under `samples/`:

- `original/`: fixtures that existed before the sample expansion.
  **Local-only, not tracked by git.**
- `synthetic/classic/`: deterministic classic EMF fixtures generated by
  `generate-synthetic-samples.mjs`.
- `synthetic/emfplus/`: deterministic EMF+ fixtures generated by the same
  script.
- `synthetic/gdiplus/`: deterministic first-party `EmfPlusDual` fixtures
  recorded by Windows `System.Drawing` using
  `scripts/generate-gdiplus-samples.ps1`.
- `real/render/`: real samples that currently parse and render without
  throwing. **Local-only, not tracked by git.**
- `real/render-gap/`: real samples that parse but expose a current renderer
  capability gap. **Local-only, not tracked by git.**
- `real/parse-gap/`: real samples that expose a current parser capability
  gap before rendering. **Local-only, not tracked by git.**

## Local-only fixtures

`samples/real/` and `samples/original/` are development-machine fixture trees
that are not distributed with the repository; they are gitignored along with
their derived darwin screenshot baselines
(`tests/visual/sample-render.spec.js-snapshots/real-*.png` and `image*.png`).
A fresh clone — including CI — sees only the `synthetic/` samples; every test
that iterates discovered fixtures shrinks accordingly, and the existence
assertions in the smoke suite are gated on the fixture tree being present.
Nothing in the npm package depends on them (`files` ships `dist/` only).

On a development machine that has the fixtures, the full sample suite and the
complete darwin snapshot baseline run exactly as before.

The `render-gap` and `parse-gap` directories are intentional. RangeError-like
failures are useful regression material, so the files stay in the sample suite
instead of being filtered out. Smoke tests document those failure modes; visual
snapshot tests only run samples outside the gap directories.

`capability-matrix.js` is the structured index for M0 sample planning. It maps
core samples to capability tags, owner milestones, gap IDs, expected behavior,
and golden-candidate status. It also records triage for every file currently in
`real/render-gap/` and `real/parse-gap/` using the stable categories `parse`,
`object`, `render`, `corrupt`, `host-limit`, and `deferred`. Keep this file in
sync whenever a gap sample moves between directories; the smoke tests verify
that each referenced sample exists and every gap file has triage.

Synthetic samples are fixed by design. They are useful for regression coverage
of specific EMF and EMF+ record families, but they do not prove general renderer
capability by themselves. Real samples sit beside them to exercise
producer-specific record ordering, nested metafiles, unusual state transitions,
and edge cases that a deterministic generator is unlikely to invent.

Regenerate synthetic samples with:

```bash
node samples/generate-synthetic-samples.mjs
```

On Windows, regenerate the native GDI+ fixtures with:

```powershell
./scripts/generate-gdiplus-samples.ps1
```

The native fixtures are excluded from the darwin screenshot tier. Their browser
output and `System.Drawing` reference output are compared by the
`windows-gdiplus` golden workflow.

Visual comparison tiers are intentionally split:

- `darwin-browser`: required Playwright browser baseline stored in
  `tests/visual/sample-render.spec.js-snapshots/`. This preserves the existing
  darwin snapshot behavior and is the default for `pnpm test:visual`.
- `windows-gdi`: optional native Windows GDI golden output stored under
  `tests/visual/goldens/windows-gdi/`.
- `windows-gdiplus`: optional native Windows GDI+ golden output stored under
  `tests/visual/goldens/windows-gdiplus/`.

Use `EMF_VISUAL_TIER=darwin-browser pnpm test:visual` for the browser baseline.
Native Windows tiers are calibration inputs, not a merge gate; helper logic
skips them automatically on non-Windows platforms or when no native golden
runner is configured.

Native golden calibration is driven by `scripts/visual-golden.js`:

```bash
pnpm golden:plan -- --tier windows-gdi
pnpm golden:verify -- --tier windows-gdi
pnpm golden:generate -- --tier windows-gdi --source-dir path/to/native-pngs
```

`golden:plan` lists the core calibration samples for the tier with their
category, capability tags, threshold group, and PNG file key. `golden:verify`
compares rendered PNGs from `tests/visual/actual/<tier>/` against the tier
golden directory and reports
sample/category/capabilities plus `changedPixelRatio`, `meanChannelDelta`, and
`maxChannelDelta` when a threshold fails. `golden:generate` imports PNGs
produced by an external native Windows runner into the tier directory when
`--source-dir` is provided. Without `--source-dir`, Windows native commands skip
with a JSON `skipReason` unless they are run on `win32` with
`EMF_NATIVE_GOLDEN_RUNNER` or `--runner` pointing at the native renderer.

Core native golden samples are intentionally smaller than the full renderable
fixture set. They are selected from `goldenCandidate` entries by tier and
capability family, then compared with separate thresholds for `text`, `bitmap`,
`gradient`, and `clip-region` so calibration failures point at the semantic
area that moved.

The local-only `real/` fixtures are collected from the public regression
suites of other EMF producers and consumers; the source name is kept in each
filename instead of a directory level, so the directory layout reflects
renderer capability status.
